# Karabiner Home Row Mods — Investigation Notes

## Setup

Home row mods (HRM) map home row keys to modifiers when held:

| Key | Modifier | Key | Modifier |
|-----|----------|-----|----------|
| `a` | left_option | `j` | right_control |
| `s` | left_command | `k` | right_shift |
| `d` | left_shift | `l` | right_command |
| `f` | left_control | `;` | right_option |

Generated by `karabiner/.config/karabiner/generate-hrm.js`.

## Current Parameters

```
to_if_alone_timeout:          400ms  — tap detection window
to_if_held_down_threshold:    300ms  — when modifier activates
simultaneous_threshold:        50ms  — window for same-hand pair detection
to_delayed_action_delay:      500ms  — (default, not set explicitly)
```

## The "Double Option" Bug

**Symptom**: Typing `as` or `;l` quickly sometimes produces ghost modifier taps instead of letters. macOS sees two rapid option taps → triggers system "double option" behavior.

**Root cause**: The cancel window between simultaneous threshold and held_down threshold was too narrow.

### How the single-key manipulator works

Each HRM key uses three Karabiner mechanisms:

- **`to_if_alone`**: Fires when key is pressed and released without any other key. Outputs the letter.
- **`to_if_held_down`**: Fires when key is held past `to_if_held_down_threshold`. Outputs the modifier.
- **`to_delayed_action.to_if_canceled`**: Fires when another key is pressed before the delayed action timer expires. Outputs the letter (the "rollover" case during fast typing).

### How simultaneous pairs interact

Simultaneous pair rules are checked **before** single-key rules. When `a` is pressed, Karabiner waits `simultaneous_threshold` ms for a second HRM key. If none arrives, it falls through to the single-key rule.

**Critical**: `to_if_held_down_threshold` counts from the original key press, NOT from when the simultaneous window expires.

### The danger zone

```
t=0          a key down
t=S          simultaneous window expires (S = simultaneous_threshold)
             └─ single-key rule takes over
             └─ to_if_held_down fires at t=H (H = held_down_threshold)
             └─ cancel window: S to H
t=H          modifier activates (if not canceled)
```

Cancel window = `held_down_threshold - simultaneous_threshold`

| Config | held_down | simultaneous | Cancel window |
|--------|-----------|--------------|---------------|
| Original | 110ms | 90ms | **20ms** |
| Current | 300ms | 50ms | **250ms** |

With 20ms, a keystroke had to land in a tiny window to cancel the modifier. With 250ms, normal fast typing easily falls within it.

## `to_delayed_action` quirks

### Default delay is 500ms

`basic.to_delayed_action_delay_milliseconds` defaults to **500ms**. This controls how long `to_if_canceled` remains active. If not set, it works independently of `to_if_held_down_threshold`.

### Timer persists past key release

The delayed action timer does NOT reset when the key is released and `to_if_alone` fires. This means a quick tap followed by another keypress within the delay window can trigger a spurious `to_if_canceled`, producing a double letter.

With default 500ms delay this doesn't seem to cause problems in practice (the window between release and re-press of the same key is typically short enough, or `to_if_alone` completing clears the state). Setting it to very large values (e.g., 5000ms for testing) causes reliable double letters.

### `to_if_canceled` requires the delay timer to be active

If `to_delayed_action_delay` expires before another key is pressed, `to_if_invoked` fires (we set it to `vk_none`) and the cancel path is closed. This was confirmed by testing: with 5000ms held_down and default 500ms delay, pressing another key at t=3000ms did NOT trigger `to_if_canceled` — the letter was swallowed.
