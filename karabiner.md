# Karabiner Home Row Mods & Symbol Layer — Investigation Notes

## Setup

Home row mods (HRM) map home row keys to modifiers when held:

| Key | Modifier | Key | Modifier |
|-----|----------|-----|----------|
| `a` | left_option | `j` | right_control |
| `s` | left_command | `k` | right_shift |
| `d` | left_shift | `l` | right_command |
| `f` | left_control | `;` | right_option |

Symbol layer (hold `g` or `h` to activate, same HRM timing):

```
LEFT HAND                                RIGHT HAND
Pinky  Ring  Mid   Idx   Idx+            Idx+  Idx   Mid   Ring  Pinky
──────────────────────────────           ──────────────────────────────
  !      @     [     ]     |               ↑     7     8     9     *
  #      $     (     )    (g)             (h)    4     5     6     +
  %      ^     {     }     ~               &     1     2     3     /
                                                       0(r_cmd)
```

`g` → `` ` `` and `h` → `↓` are reachable when the *other* trigger is held (layer mappings come before trigger rules).
`0` is mapped to `right_command`. `.` and `=` are not mapped (no available keys on Mac keyboard).

Generated by `karabiner/.config/karabiner/generate-hrm.js`.

The script generates **single-key rules only** (one per HRM key, 8 total). Simultaneous pair rules were removed to fix the double-option bug described below. Multi-modifier combos work by holding keys sequentially (hold first key until modifier activates, then hold second).

### Symbol layer implementation

The layer uses `fn` as its modifier — the same mechanism HRM uses for option/cmd/shift/ctrl. `to_if_held_down` presses `fn`, and Karabiner auto-releases it when g/h is released (standard modifier behavior). Layer mappings require `fn` in `mandatory` modifiers, so they only match when the layer is active.

Variables (`set_variable`) were tried first but don't work reliably: `to_after_key_up` only fires after `to` events, not after `to_if_held_down`, so the variable never gets cleared and the layer gets stuck.

Rule ordering in karabiner.json:
1. Manual rules (capslock, right_shift, space+jk)
2. SYM layer mappings (with `mandatory: ["fn"]`) — checked first when fn is held
3. SYM layer triggers (g, h) — HRM pattern: tap = letter, hold = fn
4. HRM modifier rules (a,s,d,f,j,k,l,;) — only reached when layer is inactive

## The "Double Option" Bug

**Symptom**: Typing `as` quickly produces ghost modifier events instead of letters. macOS sees `option DOWN, option UP, option DOWN` → triggers system "double option" behavior (e.g., opening Spotlight or input source switcher).

**Observed output** (from Karabiner EventViewer):
```
1. left_option DOWN
2. left_option UP
3. left_option DOWN
4. s DOWN (with left_option flag)
5. s UP   (with left_option flag)
6. left_option UP
```

Note: `s` appears as a **letter** (not `left_command`), and both option events come from `a`.

### Root cause: simultaneous pair rules block timer cancellation

The bug is caused by the **simultaneous pair rules** preventing `s` from reaching and canceling `a`'s `to_if_held_down` timer. It is **not** a Karabiner bug — Karabiner processes the rules exactly as configured.

#### Source code references

All paths relative to `~/workspace/third-party/karabiner/Karabiner-Elements/src/share/manipulator/`.

- `manipulator_manager.hpp` — iterates manipulators in order; `needs_wait_until_time_stamp` triggers `goto finish` (stops processing)
- `manipulators/basic/basic.hpp` — main manipulator logic; calls `to_if_held_down_->cancel()` and `to_delayed_action_->cancel()` at the **start** of `manipulate()` for every incoming event
- `manipulators/basic/to_if_held_down.hpp` — async timer via `enqueue_to_dispatcher`; `cancel()` increments id to invalidate pending callback
- `manipulators/basic/to_delayed_action.hpp` — `cancel()` posts `to_if_canceled` events immediately; uses `scoped_from_key_modifier_flags_state_restorer`
- `manipulators/basic/event_sender.hpp` — `scoped_from_key_modifier_flags_state_restorer` saves/restores modifier state around event posting

#### Event processing pipeline

From `core/CoreService/include/core_service/daemon/device_grabber.hpp`:

```
input → simple_modifications → complex_modifications → fn_function_keys → post_to_virtual_devices
```

All complex modification rules (both simultaneous pairs and single-key) live in **one** `manipulator_manager`. Events output to `complex_modifications_applied_event_queue` are **not** re-processed by complex modification rules.

#### Manipulator ordering (left hand)

Within the "HRM left" rule, manipulators are ordered:

```
 1. Pair (a,s) strict a→s     ← simultaneous pair rules first
 2. Pair (s,a) strict s→a
 3. Pair (a,d), (d,a)
 4. Pair (a,f), (f,a)
 5. Pair (s,d), (d,s)
 6. Pair (s,f), (f,s)
 7. Pair (d,f), (f,d)
 8. Single a                   ← single-key rules after
 9. Single s
10. Single d
11. Single f
```

### Detailed trace: typing `as` with gap X=120ms

Parameters: `simultaneous_threshold=90ms`, `held_down_threshold=110ms`.

#### Phase 1: `a` is processed

```
t=0ms    a key_down arrives
         → Manager iterates manipulators
         → Pair (a,s): `a` matches simultaneous set, looks for `s` in queue
           within 90ms → not found → needs_wait_until_time_stamp → goto finish
         → Event sits in queue, delayed until t=90ms

t=90ms   Simultaneous wait expires, manager re-processes `a`
         → Pair (a,s): `s` still not in queue, delay expired → is_target=false → passed
         → All other pair rules containing `a`: same result → passed
         → Single-key `a` rule: matches!
           - to_ is empty, nothing posted at key_down
           - to_if_held_down::setup(): timer via enqueue_to_dispatcher
             fires at when_now() + 110ms ≈ t=200ms
           - to_delayed_action::setup(): timer at when_now() + 500ms ≈ t=590ms
```

**Key detail**: `to_if_held_down::setup()` uses `when_now()` (real wall time when called), NOT the original event timestamp. The timer starts from when the simultaneous wait expires.

#### Phase 2: `s` arrives but gets blocked

```
t=120ms  s key_down arrives
         → Manager iterates manipulators
         → Pair (a,s): `s` matches the simultaneous `from` set
           `a` was already consumed (invalid) → not found in queue
           But `s` still matches a `from` event_definition → is_target=true
           all_from_events_found=false → needs_wait_until_time_stamp (until t=210ms)
           → goto finish
         → Manager STOPS. Single-key `a` manipulator is NEVER reached.
           a's to_if_held_down timer is NOT canceled.
```

**This is the core problem.** The pair rule claims `s` because `s` appears in its `from.simultaneous` definition, even though `a` was already consumed and the pair can never match. During the 90ms wait, `a`'s held_down timer fires unchecked.

#### Phase 3: `a`'s modifier fires

```
t=200ms  to_if_held_down timer fires (async, via dispatcher)
         → Callback checks current_held_down_id_ — still valid (not canceled)
         → Posts: left_option DOWN                              ← EVENT 1
         → post_events_at_key_down() with halt:true
           → sets halted_=true on manipulated_original_event
         → Stores left_option UP in events_at_key_up for later
```

#### Phase 4: `s`'s wait expires, delayed cancel runs

```
t=210ms  s's simultaneous wait expires, manager re-processes s
         → Pair (a,s): a not found, delay expired → is_target=false → passed
         → All other pair rules for s: same → passed
         → Single-key `a` manipulator: manipulate() called with s key_down
           At the START of manipulate() (before any target check):
             to_if_held_down_->cancel(s): increments id (too late, already fired)
             to_delayed_action_->cancel(s): posts to_if_canceled events:
```

Inside `to_delayed_action::cancel()` → `post_events()`:

The `scoped_from_key_modifier_flags_state_restorer` is an RAII object that temporarily rewinds the modifier state to what it was when the original key was pressed. This ensures `to_if_canceled` events are posted in the correct modifier context. It compares two states:
- **`key_down_modifier_flags`**: modifiers active when `a` was pressed at t=0 → **empty** (nothing held)
- **Current modifier state**: what's in the output queue's modifier_flag_manager now → **left_option DOWN** (from to_if_held_down at t=200)

These don't match, so the restorer adjusts:

```
             1. scoped_from_key_modifier_flags_state_restorer constructor:
                - Needs state to look like t=0 (no modifiers)
                - Option is down but shouldn't be → posts left_option UP    ← EVENT 2
             2. post_extra_to_events(to_if_canceled = [key_code: "a"]):
                - Checks halted_ → TRUE (set by to_if_held_down at t=200)
                - Returns immediately. Letter `a` is NOT output.
             3. scoped_from_key_modifier_flags_state_restorer destructor:
                - Undoes what constructor did → posts left_option DOWN       ← EVENT 3
```

If `halted_` had been false, the letter `a` would have been output between the option UP and option DOWN — typed cleanly without the modifier. The restorer is designed for exactly that case. But since `halted_` blocked the letter, all that remains is the empty option UP/DOWN bounce.

Then the manager continues to single-key `s` manipulator:
```
         → Single-key `s` rule: matches, sets up its own timers
           Eventually `s` is output as a letter (via to_if_alone or to_if_canceled)
           left_option is still active → s is output with option flag  ← EVENTS 4,5
```

#### Phase 5: `a` released

```
t=???    a key_up arrives
         → Single-key `a` rule handles key_up
         → post_events_at_key_up: fires stored events → left_option UP    ← EVENT 6
```

### Bug trigger condition

The bug triggers when: **typing gap X > `held_down_threshold`**

Derivation:
- `a`'s timer fires at: `simultaneous_threshold + held_down_threshold`
- `s`'s cancel arrives at: `X + simultaneous_threshold` (delayed by its own simultaneous wait)
- Bug when: `X + simultaneous_threshold > simultaneous_threshold + held_down_threshold`
- Simplifies to: **`X > held_down_threshold`**

The `simultaneous_threshold` cancels out — it delays both the timer setup AND the cancel equally. Only `held_down_threshold` determines the bug window.

| Config | held_down | Bug triggers when gap > | Typical fast typing gap |
|--------|-----------|------------------------|------------------------|
| Original (90/110) | 110ms | 110ms | 80-150ms — **often triggers** |
| "Fixed" (50/300) | 300ms | 300ms | 80-150ms — **safe** |
| Test (90/60) | 60ms | 60ms | 80-150ms — **almost always triggers** |

### Why high `held_down_threshold` is a bad fix

The modifier activates at `simultaneous_threshold + held_down_threshold` from the original keypress. With the "fixed" 50+300 config, that's **~350ms** — noticeably sluggish for intentional modifier use.

### The real fix: remove simultaneous pair rules

The pair rules are the direct cause. Without them, `s` key_down reaches single-key `a`'s `manipulate()` immediately, which calls `to_if_held_down_->cancel()` at the start — no 90ms blockage. This allows a much lower `held_down_threshold` (fast modifier activation) without triggering the bug.

Tradeoff: lose same-hand modifier combos (e.g., hold `a+s` for option+command). Cross-hand combos (e.g., `a+k` for option+shift) are unaffected since they don't share simultaneous pair rules.

## `to_delayed_action` quirks

### Default delay is 500ms

`basic.to_delayed_action_delay_milliseconds` defaults to **500ms**. This controls how long `to_if_canceled` remains active. If not set, it works independently of `to_if_held_down_threshold`.

### `to_if_canceled` requires the delay timer to be active

If `to_delayed_action_delay` expires before another key is pressed, `to_if_invoked` fires (we set it to `vk_none`) and the cancel path is closed.

### `halted` flag prevents double output

When `to_if_held_down` fires with `halt: true`, it sets `halted_=true` on the `manipulated_original_event`. Subsequent `post_extra_to_events` calls (including from `to_if_canceled`) check this flag and return early. This prevents the letter from being output after the modifier — but the `scoped_from_key_modifier_flags_state_restorer` still runs, producing the option UP/DOWN bounce that macOS interprets as a double-option tap.
